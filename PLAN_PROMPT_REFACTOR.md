- OKR.md 的所有内容都会加载到提示词中

### 4. SENSE 方法整合 ✅

**核心思想**:
```
问题澄清 → 形成视角 → 识别矛盾 → (困境时逆转) → 批判检查
```

**每个步骤的指导**:

#### S - Subject (澄清问题)
- 还原为"现状-期望-差距"三要素
- 分解事实性模糊和价值性模糊
- 持续追问直到清晰

#### E - Perspective (形成视角)
- 绘制概念地图
- 给出简单清晰的假设
- 融贯性检验（自洽、他洽、续洽）

#### N - Judgment (识别主要矛盾)
- 绘制系统循环图
- 识别主要矛盾（全局性、根本性、决定性）
- 选择最优控制手段

#### S - Reverse (辩证逆转)
- 仅在困境时使用
- 核心问题："如果 A 是 B 的存在条件，那么 B 应当如何？"
- 在更高系统层次找到创新解法

#### E - Critique (批判性检查)
- 检查逻辑有效性
- 保持良质标准
- 必要时切换系统层次

### 5. 四大行为约束 ✅

#### 约束 1: 追问澄清需求

**核心**: 不要立即开始规划！

**要求**:
- 先追问至少 3-5 个问题
- 使用 SENSE-Subject 方法还原"现状-期望-差距"
- 确认现状、期望、约束、优先级

**追问示例**:
- "你说的 XXX 功能，具体是指...吗？"
- "当前系统在这方面的状态是怎样的？"
- "你期望这个功能达到什么效果？有具体的指标吗？"

#### 约束 2: 先获取事实再判断

**核心**: 使用所有可能的工具获取事实性信息

**工具清单**:
- Read 工具：阅读代码、文档、配置
- Grep 工具：搜索实现、API 使用
- Bash 工具：检查系统状态、环境
- WebSearch 工具：搜索最佳实践、文档

**要求**:
- 列出获取到的关键事实
- 说明事实如何影响方案选择
- 明确信息缺口

#### 约束 3: 模块化分解

**核心**: 每个 task 必须可以独立验证

**要求**:
- **独立性**: 可以独立开发、测试、验证
- **完整性**: 是完整的功能单元，不是半成品
- **可验证性**: 有明确的验收标准和测试方法
- **合理粒度**: 0.5-2 天的工作量

**不良示例**:
```
Task1: 写代码
Task2: 测试
Task3: 部署
```
（问题：流程化，不是模块化）

**良好示例**:
```
Task1: 实现用户认证模块
Task2: 实现权限管理模块
Task3: 实现用户信息管理模块
```

#### 约束 4: 严格检查依赖关系

**核心**: 依赖关系决定后续工作是否顺利

**检查清单**:
- [ ] 每个 task 的依赖都已明确列出
- [ ] 不存在循环依赖
- [ ] 依赖关系是技术必要的
- [ ] 关键路径上的任务数量已最小化
- [ ] 可以绘制出清晰的任务依赖图

## 二、完整工作流程

### 阶段 1: 需求澄清（必须完成）
1. 阅读用户需求
2. 提出 3-5 个追问问题
3. 等待用户回答
4. 还原为"现状-期望-差距"三要素

### 阶段 2: 信息收集（必须完成）
1. 使用工具获取事实信息
2. 列出关键事实清单
3. 识别信息缺口

### 阶段 3: 系统化思考
1. 应用 SENSE 方法
2. 输出思考过程（可选）

### 阶段 4: 任务分解
1. 按模块化原则分解
2. 定义目标、关键结果、测试方法
3. 检查依赖关系

### 阶段 5: 输出任务列表
1. 按 task.md 格式输出
2. 建议执行顺序
3. 预估时间和关键路径
