# Plan 提示词改造：项目类型判断与 OKR/SPEC 初始化

## 改造目标

在 Plan 提示词中添加对新项目初始化和旧项目探索的处理逻辑，确保在任何情况下都能获得明确的 OKR 和 SPEC。

## 核心理念

- **OKR** = 目标 (Objectives) + 达成目标的关键成果 (Key Results)
- **SPEC** = 做事的方法和技术规范 (Specifications)
- **原则**：没有明确的 OKR 和 SPEC，不要开始任务规划

## 改造内容

### 1. 新增章节：三.1 项目类型判断与处理策略

添加了完整的项目类型判断逻辑和三种处理策略：

#### 情况 A: 新项目初始化（OKR 和 SPEC 不存在）

**识别标志**：
- OKR 内容为 "暂无项目 OKR 信息"
- SPEC 内容为 "暂无项目 SPEC 信息"
- 工作目录下文件很少或为空

**处理策略**：

1. **引导用户完成 OKR 设想**
   - 通过对话引导用户思考项目目标
   - 追问核心目标、成功标准、关键结果、目标用户、优先级
   - 提供 OKR 模板建议

2. **引导用户完成 SPEC 设想**
   - 引导用户思考做事方法和技术规范
   - 追问技术栈、架构风格、代码规范、工程实践、质量要求
   - 提供 SPEC 模板建议

3. **完成后再进行任务规划**
   - 总结并确认用户的 OKR 和 SPEC
   - 建议保存到 `.rick/OKR.md` 和 `.rick/SPEC.md`
   - 基于确认的 OKR 和 SPEC 进行任务规划

#### 情况 B: 旧项目迭代（工作目录存在文件但无 OKR/SPEC）

**识别标志**：
- OKR 或 SPEC 不存在
- 但工作目录下存在代码文件、配置文件、文档等

**处理策略**：

1. **探索项目结构**
   - 查看项目根目录结构
   - 查看 README 或文档
   - 查看主要代码目录
   - 查看配置文件
   - 查看测试文件

2. **理解项目的 OKR**
   - 从 README.md 推断项目描述和目标
   - 从主要功能模块推断核心功能
   - 从测试用例推断验收标准
   - 从 Issue/TODO 推断待完成目标
   - 输出理解结果

3. **理解项目的 SPEC**
   - 从依赖文件推断技术栈
   - 从目录结构和代码组织推断架构设计
   - 从现有代码推断代码风格
   - 从测试文件推断测试策略
   - 从 .git, CI 配置推断工程实践
   - 输出理解结果

4. **与用户确认理解**
   - 展示对项目 OKR 和 SPEC 的理解
   - 询问是否准确，是否需要补充或修正
   - 建议保存到 `.rick/OKR.md` 和 `.rick/SPEC.md`

5. **完成确认后再进行任务规划**

#### 情况 C: 正常迭代（OKR 和 SPEC 都存在）

直接进入需求澄清阶段。

### 2. 修改工作流程：新增阶段 0

在原有的工作流程前增加"阶段 0: 项目类型判断"：

```
阶段 0: 项目类型判断（必须首先完成）
├─ 检查 OKR 和 SPEC 状态
├─ 检查工作目录状态
└─ 判断项目类型并选择策略
   ├─ 情况 A: 新项目初始化 → 引导用户完成 OKR/SPEC 设想
   ├─ 情况 B: 旧项目迭代 → 探索并理解 OKR/SPEC
   └─ 情况 C: 正常迭代 → 直接进入阶段 1
```

### 3. 修改"开始执行"部分

强调项目类型判断是第一步：

```
1. 第一步：判断项目类型
2. 如果是新项目初始化：引导用户明确 OKR 和 SPEC
3. 如果是旧项目迭代：探索并理解 OKR 和 SPEC
4. 如果是正常迭代：直接进行需求澄清

记住：
- 项目类型判断是第一步，也是最重要的一步！
- 没有明确的 OKR 和 SPEC，不要开始任务规划！
- 追问和探索是规划的基础，不要跳过！
```

### 4. 修改质量检查清单

添加项目类型判断的检查项：

```
项目类型判断：
- [ ] 已检查 OKR 和 SPEC 的状态
- [ ] 已检查工作目录的状态
- [ ] 已明确项目类型（新项目/旧项目/正常迭代）
- [ ] 如果是新项目：已引导用户完成 OKR 和 SPEC 设想
- [ ] 如果是旧项目：已探索并理解项目的 OKR 和 SPEC
- [ ] 已确保 OKR 和 SPEC 明确后才开始任务规划
```

## 关键设计原则

### 新项目初始化
- ✅ 不要假设用户的 OKR 和 SPEC，必须通过对话引导
- ✅ 对话要循序渐进，先 OKR（目标）再 SPEC（方法）
- ✅ 每个问题都要等待用户回答，不要一次性问太多
- ✅ 提供模板建议，帮助用户快速上手

### 旧项目迭代
- ✅ 不要假设项目的 OKR 和 SPEC，必须通过探索理解
- ✅ 探索要全面，使用所有可用工具（Read, Grep, Bash, Glob 等）
- ✅ 理解要准确，基于事实而非猜测
- ✅ 必须与用户确认理解，不要自作主张

### 共同原则
- ✅ OKR 是目标与达成目标的关键成果
- ✅ SPEC 是做事的方法和技术规范
- ✅ 没有明确的 OKR 和 SPEC，不要开始任务规划

## 使用示例

### 场景 1: 新项目初始化

用户执行：
```bash
rick plan "创建一个 TODO 应用"
```

Claude 的行为：
1. 检查 OKR/SPEC 状态 → 发现都不存在
2. 检查工作目录 → 发现基本为空
3. 判断为"新项目初始化"
4. 开始引导用户：
   ```
   我注意到这是一个新项目，让我先帮你明确项目的 OKR（目标与关键结果）：

   1. 这个 TODO 应用的核心目标是什么？你希望通过这个项目解决什么问题？
   2. 成功的标准是什么？如何衡量这个项目是否达成了目标？
   3. 项目的关键结果有哪些？比如功能完成度、性能指标、用户体验等
   ...
   ```

### 场景 2: 旧项目迭代

用户在已有项目中执行：
```bash
rick plan "添加用户认证功能"
```

Claude 的行为：
1. 检查 OKR/SPEC 状态 → 发现都不存在
2. 检查工作目录 → 发现存在代码文件
3. 判断为"旧项目迭代"
4. 开始探索项目：
   ```bash
   ls -la
   cat README.md
   find . -type f -name "*.go" | head -20
   cat go.mod
   ```
5. 输出理解结果：
   ```
   基于项目探索，我理解的项目 OKR 是：

   目标 1: 构建一个轻量级的任务管理系统
   - 关键结果1: 支持任务的增删改查
   - 关键结果2: 支持任务的优先级和标签管理

   项目 SPEC 是：

   技术栈: Go + Gin + SQLite
   架构设计: MVC 三层架构
   ...

   请确认我的理解是否准确？
   ```

### 场景 3: 正常迭代

用户在已有 OKR/SPEC 的项目中执行：
```bash
rick plan "优化数据库查询性能"
```

Claude 的行为：
1. 检查 OKR/SPEC 状态 → 发现都存在
2. 判断为"正常迭代"
3. 直接进入需求澄清：
   ```
   让我先澄清一下需求：

   1. 你提到的数据库查询性能问题，具体是指哪些查询？
   2. 当前的性能指标是什么？期望达到什么水平？
   3. 有没有观察到具体的慢查询日志或性能瓶颈？
   ...
   ```

## 预期效果

### 对新项目
- ✅ 不会直接开始规划任务
- ✅ 会引导用户思考和明确 OKR 和 SPEC
- ✅ 提供模板和示例，降低用户门槛
- ✅ 确保项目从一开始就有清晰的目标和规范

### 对旧项目
- ✅ 不会假设项目的目标和规范
- ✅ 会主动探索项目结构和代码
- ✅ 会输出理解结果并与用户确认
- ✅ 确保规划任务时有准确的上下文

### 对所有项目
- ✅ 强制执行"先明确 OKR/SPEC，再规划任务"的原则
- ✅ 避免盲目规划导致的方向偏差
- ✅ 提高任务规划的质量和准确性
- ✅ 建立项目的知识积累基础

## 总结

这次改造的核心价值：

1. **强制 OKR/SPEC 初始化**：无论新旧项目，都必须有明确的 OKR 和 SPEC
2. **智能项目类型判断**：自动识别项目状态，选择合适的处理策略
3. **引导式对话**：对新项目，通过追问引导用户思考
4. **探索式理解**：对旧项目，通过工具探索并理解现状
5. **确认式规划**：始终与用户确认理解后再开始规划

这确保了 Rick 的 Context-First 理念得到彻底贯彻：**没有上下文，就没有规划**。
