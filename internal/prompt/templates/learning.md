### 输出目录结构

```
.rick/jobs/{{job_id}}/learning/
├── SUMMARY.md              # 执行总结报告（必需）
├── OKR_UPDATE.md           # OKR 更新建议（如需要）
├── SPEC_UPDATE.md          # SPEC 更新建议（如需要）
├── wiki/                   # Wiki 文档更新（如需要）
│   ├── <主题1>.md
│   └── <主题2>.md
└── skills/                 # 可复用技能提取（如需要）
    ├── <技能1>.md
    └── <技能2>.md
```

### 文件生成规则

1. **SUMMARY.md**（必需生成）
   - 执行概述和关键成就
   - 问题与教训分析
   - 技术总结和建议
   - 知识沉淀清单

2. **OKR_UPDATE.md**（按需生成）
   - 仅当需要更新项目目标时生成
   - 包含具体的更新建议和理由
   - 标注新增/修改/删除的部分

3. **SPEC_UPDATE.md**（按需生成）
   - 仅当需要更新技术规范时生成
   - 包含具体的更新建议和理由
   - 标注新增/修改/删除的部分

4. **wiki/<主题>.md**（按需生成）
   - 为新概念、新技术或重要模式创建 wiki 文档
   - 每个主题一个独立文件
   - 文件名使用小写字母和下划线（如：error_handling.md）

5. **skills/<技能>.md**（按需生成）
   - 提取可复用的技能模式
   - 包含使用场景、示例代码和注意事项
   - 文件名使用小写字母和下划线（如：retry_mechanism.md）

在生成 OKR_UPDATE.md 和 SPEC_UPDATE.md 时，必须参考以下格式（来自 plan.md）：

### OKR 格式参考（来自 plan.md）

```markdown
# 项目目标 (Objectives)

## O1: [核心目标1]
描述项目要达成的核心目标

### 关键结果 (Key Results)
- KR1.1: [可衡量的结果1]
- KR1.2: [可衡量的结果2]
- KR1.3: [可衡量的结果3]

## O2: [核心目标2]
```

**关键点**:
- 目标编号: O1, O2, O3...

- 必须有 "关键结果 (Key Results)" 标题
- 关键结果必须可衡量

## 质量标准

- 学习总结应该全面而深入

## 工作流程

1. **分析阶段**
   - 使用 `git show <commit_hash>` 查看每个任务的代码变更
   - 阅读 debug.md 了解遇到的问题和解决方法
   - 识别关键洞察、模式和改进点

2. **生成阶段**
   - 在 `.rick/jobs/{{job_id}}/learning/` 目录下生成所有文档
   - 确保 SUMMARY.md 必需生成
   - 按需生成 OKR_UPDATE.md、SPEC_UPDATE.md、wiki/、skills/

3. **审核阶段**（由人工完成）
   - 审核生成的所有文档
   - 决定是否接受更新建议
   - 手动将更新合并到 `.rick/` 目录

## ⚠️ 重要约束

1. **禁止直接修改项目文档**: 不要直接修改 `.rick/OKR.md`、`.rick/SPEC.md`、`.rick/wiki/`、`.rick/skills/`
2. **所有输出在 learning 目录**: 所有生成的文档必须在 `.rick/jobs/{{job_id}}/learning/` 下
3. **生成更新建议而非直接修改**: 使用 `XXX_UPDATE.md` 格式提供更新建议
4. **清晰的合并指令**: 在每个更新建议文档中提供清晰的合并指令
5. **完整的 SUMMARY.md**: 必须生成完整的执行总结报告
6. **⭐ 格式一致性（CRITICAL）**: OKR_UPDATE.md 和 SPEC_UPDATE.md 必须严格遵循 `plan.md` 中定义的格式规范
   - OKR 格式: O1/O2/O3 编号 + KR1.1/KR1.2 编号 + "关键结果 (Key Results)" 标题
   - SPEC 格式: 技术栈/架构设计/开发规范/工程实践 四个类别 + `- 项目名: 内容` 格式
   - 这是为了确保生成的更新建议可以无缝合并到项目文档中
