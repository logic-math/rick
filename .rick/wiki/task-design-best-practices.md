## 核心原则

### 1. SMART 任务原则

每个任务应该满足 SMART 标准：

- **Specific（具体）**: 明确的单一目标
- **Measurable（可衡量）**: 清晰的成功标准
- **Achievable（可实现）**: 5-10 分钟可完成
- **Relevant（相关）**: 与项目目标一致
- **Time-bound（有时限）**: 明确的执行时间范围

### 2. 单一职责原则

每个任务只做一件事：

```markdown
✅ 好的任务: "生成 OKR.md 文档"
❌ 坏的任务: "生成 OKR.md 和 SPEC.md 文档"

✅ 好的任务: "创建 Wiki 索引和架构文档"
❌ 坏的任务: "创建完整的 Wiki 知识库"
```

### 3. 最小化依赖原则

- 尽量减少任务间的依赖关系
- 无依赖的任务可以并行执行
- 依赖关系应该是逻辑必需的，而非人为强加的

## 任务分解

### 最佳粒度：5-10 分钟

**经验数据**（来自 Job 0）:
- 7 个任务，总执行时间 ~47 分钟
- 平均每个任务 6.7 分钟
- 零重试率

**粒度判断标准**:

| 任务复杂度 | 预估时间 | 建议 |
|-----------|---------|------|
| 单一文档生成 | 5-10 分钟 | ✅ 合适 |
| 多个文档生成 | 15-20 分钟 | ⚠️ 考虑拆分 |
| 复杂系统实现 | 30+ 分钟 | ❌ 必须拆分 |

### 分解策略

#### 策略 1：按阶段分解

```
大任务: "构建完整的全局上下文"
↓
Phase 1: 生成标准文档（OKR, SPEC）
  - task1: 生成 OKR.md
  - task2: 生成 SPEC.md
Phase 2: 生成结构化文档（Wiki）
  - task3: 创建 Wiki 索引
  - task4: 创建模块文档
Phase 3: 提取可复用内容（Skills）
  - task5: 提取技能库
Phase 4: 验证完整性
  - task6: 验证并生成报告
```

#### 策略 2：按模块分解

```
大任务: "实现用户认证系统"
↓
task1: 实现用户注册接口
task2: 实现用户登录接口
task3: 实现 JWT 令牌管理
task4: 实现权限验证中间件
task5: 编写单元测试
```

#### 策略 3：按依赖分解

```
大任务: "优化数据库查询性能"
↓
task1: 分析慢查询日志（无依赖）
task2: 设计索引优化方案（依赖 task1）
task3: 实施索引优化（依赖 task2）
task4: 验证性能提升（依赖 task3）
```

## 依赖关系设计

### 三种依赖模式

#### 1. 并行模式（无依赖）

适用于独立的基础任务：

```
task1: 生成 OKR.md     (无依赖)
task2: 生成 SPEC.md    (无依赖)
```

**优点**: 可以并行执行，提高效率
**适用场景**: 基础文档生成、独立模块实现

#### 2. 串行模式（线性依赖）

适用于需要前置输出的任务：

```
task1: 生成 OKR.md
  ↓
task3: 创建 Wiki 索引（依赖 task1，使用 OKR 作为标准）
  ↓
task5: 提取技能库（依赖 task3，从 Wiki 提取）
```

**优点**: 确保上下文传递，保证一致性
**适用场景**: 需要前置输出作为输入的任务

#### 3. 汇聚模式（多重依赖）

适用于验证或集成任务：

```
task1, task2, task3, task4, task5
  ↓   ↓   ↓   ↓   ↓
task6: 验证完整性（依赖所有前置任务）
```

**优点**: 确保所有前置任务完成后再验证
**适用场景**: 完整性验证、集成测试、总结报告

### 依赖设计原则

1. **最小依赖**: 只添加逻辑必需的依赖
2. **明确依赖**: 在 task.md 的"依赖关系"部分明确声明
3. **避免循环**: 绝不能出现循环依赖
4. **扁平优先**: 尽量减少依赖层级深度

## 上下文传递

### 上下文传递策略

#### 策略 1：标准先行

先定义标准文档，后续任务遵循标准：

```
task1: 生成 OKR.md（定义项目目标和标准）
  ↓ 上下文传递
task3: 创建 Wiki 索引（遵循 OKR 定义的标准）
```

**实践经验**（Job 0）:
- OKR 定义了 5 个核心目标
- 后续 Wiki 文档都围绕这 5 个目标组织
- 确保了文档的一致性和完整性

#### 策略 2：统一模板

所有同类文档使用相同的结构：

```
task3: 创建 Wiki 索引（定义文档模板）
  ↓ 模板传递
task4: 创建模块文档（遵循统一模板）
```

**模板示例**:
```markdown
# 模块名称

## 概述
## 核心类型
## 主要函数
## 使用示例
## 常见问题
## 相关模块
```

#### 策略 3：自动验证

最后包含验证任务，确保一致性：

```
task1-5: 生成各类文档
  ↓ 验证上下文
task6: 验证完整性（检查一致性、完整性、链接有效性）
```

**验证维度**:
- 文件存在性
- 文件完整性（非空、结构）
- 内容一致性（OKR vs SPEC vs Wiki vs Skills）
- 链接有效性
- 关键词覆盖

## 验证标准

### 任务验证的三个层次

#### 1. 输出验证（Output Validation）

验证任务是否产生了预期的输出：

```markdown
# 测试方法

1. 检查文件是否存在
   ```bash
   test -f .rick/OKR.md || exit 1
   ```

2. 检查文件非空
   ```bash
   test -s .rick/OKR.md || exit 1
   ```

3. 检查文件结构
   ```bash
   grep -q "^# Rick CLI - OKR" .rick/OKR.md || exit 1
   ```
```

#### 2. 质量验证（Quality Validation）

验证输出的质量是否符合标准：

```markdown
# 测试方法

1. 检查文档长度（≥ 400 行）
   ```bash
   lines=$(wc -l < .rick/OKR.md)
   test $lines -ge 400 || exit 1
   ```

2. 检查关键章节存在
   ```bash
   grep -q "## 核心目标" .rick/OKR.md || exit 1
   grep -q "## 关键结果" .rick/OKR.md || exit 1
   ```

3. 检查关键词覆盖
   ```bash
   grep -c "Context-First" .rick/OKR.md | grep -q "[3-9]" || exit 1
   ```
```

#### 3. 集成验证（Integration Validation）

验证输出与其他组件的集成：

```markdown
# 测试方法

1. 检查与 SPEC 的一致性
   ```bash
   # 检查 OKR 中的目标是否在 SPEC 中有对应实现
   grep "构建 Context-First 框架" .rick/OKR.md
   grep "Context-First" .rick/SPEC.md || exit 1
   ```

2. 检查与代码的一致性
   ```bash
   # 检查 OKR 中提到的模块是否存在
   test -d internal/prompt || exit 1
   ```
```

## 实战案例：Job 0 分析

### 任务分解分析

Job 0 的 7 个任务是任务设计的教科书范例：

| Task | 名称 | 粒度 | 依赖 | 结果 |
|------|------|------|------|------|
| task1 | 生成 OKR.md | 5 分钟 | 无 | ✅ 一次通过 |
| task2 | 生成 SPEC.md | 8 分钟 | 无 | ✅ 一次通过 |
| task3 | 创建 Wiki 索引 | 8 分钟 | task1 | ✅ 一次通过 |
| task4 | 创建模块文档 | 6 分钟 | task2 | ✅ 一次通过 |
| task5 | 提取技能库 | 12 分钟 | task3,4 | ✅ 一次通过 |
| task6 | 创建示例文档 | 8 分钟 | task1-5 | ✅ 一次通过 |
| task7 | 验证完整性 | 5 分钟 | task1-6 | ✅ 一次通过 |

### 成功因素分析

#### 1. 合理的粒度

- 平均 6.7 分钟/任务
- 最短 5 分钟，最长 12 分钟
- 所有任务都在 5-15 分钟范围内

#### 2. 清晰的依赖关系

```
     task1 (OKR)        task2 (SPEC)
        ↓                    ↓
     task3 (Wiki索引)    task4 (模块文档)
        ↓                    ↓
        └────────┬───────────┘
                 ↓
            task5 (Skills)
                 ↓
            task6 (示例)
                 ↓
            task7 (验证)
```

**依赖设计亮点**:
- task1 和 task2 并行（无依赖）
- task3 依赖 task1（需要 OKR 作为标准）
- task4 依赖 task2（需要 SPEC 作为规范）
- task5 汇聚 task3 和 task4（提取技能）
- task7 汇聚所有任务（完整性验证）

#### 3. 良好的上下文传递

- **OKR 先行**: task1 定义了 5 个核心目标，后续任务都围绕这些目标
- **SPEC 规范**: task2 定义了技术规范，task4 的模块文档遵循规范
- **统一模板**: task3 定义了 Wiki 结构，task4 遵循统一模板
- **自动验证**: task7 验证了所有文档的一致性

#### 4. 明确的验证标准

每个任务都有清晰的验证标准：

**task1 验证**:
```bash
# 1. 文件存在
test -f .rick/OKR.md

# 2. 文档长度 ≥ 400 行
lines=$(wc -l < .rick/OKR.md)
test $lines -ge 400

# 3. 包含 5 个核心目标
grep -c "^### O[1-5]:" .rick/OKR.md | grep -q "5"

# 4. 每个目标有 5 个关键结果
grep -c "^- \*\*KR" .rick/OKR.md | grep -q "[2-3][0-9]"
```

## 常见反模式

### ❌ 反模式 1：任务过大

**问题**:
```markdown
# 任务名称
实现完整的用户管理系统

# 任务目标
实现用户注册、登录、权限管理、个人信息管理等功能
```

**为什么不好**:
- 任务太大，预估时间 > 30 分钟
- 包含多个目标，违反单一职责原则
- 难以验证，难以重试

**改进方案**:
```markdown
拆分为 5 个任务：
- task1: 实现用户注册接口
- task2: 实现用户登录接口
- task3: 实现 JWT 令牌管理
- task4: 实现权限验证中间件
- task5: 实现个人信息管理接口
```

### ❌ 反模式 2：依赖关系不明确

**问题**:
```markdown
# 依赖关系
（空）

# 任务名称
创建模块文档

# 任务目标
为所有核心模块创建 Wiki 文档，遵循 SPEC 规范
```

**为什么不好**:
- 任务描述中提到"遵循 SPEC 规范"，但依赖关系中没有声明
- 可能导致任务执行时缺少必要的上下文

**改进方案**:
```markdown
# 依赖关系
task2

# 任务名称
创建模块文档

# 任务目标
为所有核心模块创建 Wiki 文档，遵循 SPEC.md 中定义的规范
```

### ❌ 反模式 3：验证标准模糊

**问题**:
```markdown
# 测试方法
1. 检查文件是否创建成功
2. 检查内容是否正确
```

**为什么不好**:
- "创建成功"和"内容正确"太模糊
- 无法自动化验证
- 容易产生歧义

**改进方案**:
```markdown
# 测试方法
1. 检查文件存在
   ```bash
   test -f .rick/OKR.md || exit 1
   ```

2. 检查文件非空
   ```bash
   test -s .rick/OKR.md || exit 1
   ```

3. 检查文档结构
   ```bash
   grep -q "^# Rick CLI - OKR" .rick/OKR.md || exit 1
   grep -c "^### O[1-5]:" .rick/OKR.md | grep -q "5" || exit 1
   ```

4. 检查文档长度 ≥ 400 行
   ```bash
   lines=$(wc -l < .rick/OKR.md)
   test $lines -ge 400 || exit 1
   ```
```

### ❌ 反模式 4：循环依赖

**问题**:
```markdown
task1 依赖 task2
task2 依赖 task3
task3 依赖 task1  ← 循环依赖！
```

**为什么不好**:
- 无法确定执行顺序
- DAG 拓扑排序会失败
- 逻辑上不合理

**改进方案**:
```markdown
重新设计依赖关系，打破循环：
task1 (无依赖)
task2 依赖 task1
task3 依赖 task2
```

## 检查清单

### 任务定义检查清单

在创建 task.md 之前，检查以下项目：

- [ ] **任务名称**: 简洁明确（≤ 10 个字）
- [ ] **任务目标**: 单一、具体、可衡量
- [ ] **任务粒度**: 预估 5-10 分钟可完成
- [ ] **依赖关系**: 明确声明，无循环依赖
- [ ] **关键结果**: 3-5 个可验证的结果
- [ ] **测试方法**: 自动化验证脚本

### 任务执行检查清单

在执行任务之前，检查以下项目：

- [ ] **前置任务**: 所有依赖任务已完成
- [ ] **上下文完整**: 必要的上下文文件已存在
- [ ] **工具就绪**: 所需工具已安装
- [ ] **验证脚本**: 测试脚本可执行

### 任务完成检查清单

在标记任务完成之前，检查以下项目：

- [ ] **输出验证**: 所有预期输出已生成
- [ ] **质量验证**: 输出质量符合标准
- [ ] **集成验证**: 与其他组件集成正常
- [ ] **Git 提交**: 代码已提交到 Git
- [ ] **文档更新**: 相关文档已更新

---

## 总结

### 核心要点

1. **任务粒度**: 5-10 分钟，单一目标
2. **依赖设计**: 最小化依赖，避免循环
3. **上下文传递**: 标准先行 → 统一模板 → 自动验证
4. **验证标准**: 输出验证 + 质量验证 + 集成验证

### 成功公式

```
清晰任务定义 + 合理粒度 + 良好上下文传递 = 零重试率
```
