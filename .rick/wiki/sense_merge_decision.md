# rick 与 sense 的架构决策：将 sense 合并为 rick 的上下文引擎

> **文档类型：** SENSE 思考记录
> **创建时间：** 2026-04-01
> **思考主体：** sunquan
> **适用场景：** 作为下次会话的上下文，供人和 AI 共读

---

## 1. 澄清问题（Subject）

**现状**
同时维护两个 AI coding 框架：rick 和 sense。rick 是程序主控模式，sense 是 AI main agent 主控模式。之前凭感觉维护两套，没有理性判断过哪个更值得深耕。今天和同事交流后产生了深刻的想法，开始认真思考这个问题。

**期望**
把精力集中在一个框架上继续深耕，做出一个能长期演进的 AI coding 框架。

**差距**
缺少一个清晰的判断依据。之前没有仔细思考，只是凭感觉认为"一个 agent 进行交互看起来才是终局"，但这个感觉没有经过理性检验。

---

## 2. 假设视角（Perspective）

**核心假设**
main agent 模式破坏上下文在项目周期中的一致性，而上下文一致性是 hardness 工程的第一性原理。

**假设依据**

1. **调度逻辑应该是确定性程序**：反馈循环本身是一个确定性的程序流程，不需要每次都通过 token 去执行。用 AI 驱动调度会引入不确定性，且消耗 token，没有带来额外收益。

2. **main agent 上下文压缩问题**：main agent 的上下文窗口消耗光后，Claude Code 不会自动清理，只会压缩。压缩可能丢失记忆，丢失记忆后整个 loop 就起不来了。

3. **对话框的价值悖论**：sense 的核心价值是可以打断干预，但打断会破坏上下文生命周期。如果不打断，对话框和后台运行没有区别，唯一区别是 main agent 要额外消耗 token 驱动。

4. **hardness 的本质**：hardness 本质上是模块化上下文，对上下文的管理有两个方面：（1）将确定性的事物沉淀为工具；（2）将所有上下文模块化并管理其生命周期。对话框形式在 doing 时引入的上下文无法被管理，破坏了模块化上下文的一致性。

**融贯性检验**
- 自洽：AI 生成 + 程序验证 = 确定性。rick 的模式（AI 生成 plan，程序 check 验证）满足这个原则。✓
- 他洽：sense 也有 plan_check/doing_check/learning_check，但核心问题不在于有没有验证，而在于调度层本身引入了不确定性（main agent 上下文管理问题）。✓
- 续洽：如果上下文一致性是第一性原理，那 Claude Code 本身未来也会往静默执行方向发展（预判：Claude Code 会发展为可静默执行的 CLI，缺的只是 task 调度层）。✓

## 3. 矛盾判断（Judgment）

**系统中的主要矛盾**
上下文一致性（rick 的核心）和可观测性/灵活调度（sense 的核心）之间的矛盾。sense 的 main agent 模式为了获得灵活调度和可观测性，牺牲了上下文在项目周期中的一致性。

**矛盾分析**
- 上下文一致性是根本：没有一致性，生产级代码输出无法保证
- 可观测性是次要的：可以通过定制化 agent 或日志方式实现，不是 main agent 模式独有的优势
- sense 的唯一不可替代优势（对话框交互感）本质上是产品形态上的"感觉优雅"，而非工程上的实质优势

**选择的控制手段**
将 sense 合并到 rick 中作为其中一个模块，未来持续迭代为其上下文引擎，解决上下文一致性与灵活性的矛盾。

**排除的选项**
- 排除继续独立维护 sense：sense 的核心设计模式（main agent 主控）与上下文一致性原则相悖，没有不可替代的价值
- 排除 sense 的 ai-loop skill：ai-loop 是 AI 全自动循环，属于 main agent 模式的核心体现，与决策方向矛盾
- 保留 sense 的 human-loop skill：human-loop 逼迫人类深度思考，符合 rick 的设计初衷，与 rick 的 OKR 保持一致

---

## 4. 解决方法（Reverse）

> 注：Judgment 阶段已找到解决方案，此部分记录常规路径。

**方案描述**
1. 废弃 sense 作为独立框架
2. 将 sense 的 human-loop skill 合并进 rick
3. sense 继续迭代，但作为 rick 的上下文引擎模块存在，不作为独立产品

**关键路径**
1. 把 human-loop skill 迁移到 rick 的 skills 目录
2. sense 的 CLI 工具（init/plan/doing/learning）评估是否有增量价值，按需合并
3. 未来 sense 迭代方向：作为 rick 的上下文引擎，丰富上下文管理能力
4. 长期目标：构建一个 hardness 优先的专用 agent，支持 rick 的主控逻辑

**新发现的可控变量**
无逆转触发。

## 5. 过程批判（Critique）

**核心假设清单**
| 假设 | 如果不成立，结论会…… |
|------|---------------------|
| 上下文一致性是 hardness 工程的第一性原理 | 需要重新评估 main agent 模式的价值，可能两种模式都有各自适用场景 |
| main agent 上下文压缩会导致 loop 崩溃 | 如果 Claude Code 的压缩能力足够强，sense 模式的核心缺陷消失，结论可能反转 |
| 可观测性可以通过日志/定制化 agent 实现 | 如果对话框交互有不可替代的价值（如外部用户体验），废弃 sense 可能是错误的 |

**逻辑漏洞**

- sense 的 ai-loop 被排除，但其底层的 DAG 调度、tasks.json 状态管理等机制是否有价值，未单独评估。

**良质确认**
用户确认这个方案符合自己真正想要的：rick 的设计初衷是"上下文优先"，human-loop 能逼迫人类深度思考，与这个初衷一致。长期目标是做一个 hardness 优先的专用 agent，这个方向也与 rick 的演进路径吻合。
