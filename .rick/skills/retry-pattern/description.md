# 失败重试模式技能

## 技能概述

失败重试模式（Retry Pattern）是一种处理瞬态故障的弹性设计模式，当操作失败时自动重试，并在每次重试之间记录详细的调试信息。本技能结合了重试机制、错误分析和上下文累积，使 AI Agent 能够从失败中学习并逐步改进。

核心特点：**失败即学习**。每次失败都会生成结构化的调试日志（debug.md），下一轮重试会加载这些上下文，让 AI Agent 理解之前的失败原因并调整策略。

## 使用场景

### 1. AI Agent 任务执行
- **场景**: AI Agent 执行复杂任务（如代码生成、测试验证）时可能失败
- **示例**: Rick CLI 使用 Claude Code 生成代码，测试未通过需要重试
- **价值**: 自动重试并累积失败经验，提高成功率

### 2. 网络请求重试
- **场景**: 网络不稳定导致 API 调用失败
- **示例**: 调用第三方服务超时，需要自动重试
- **价值**: 提高系统可用性，减少人工干预

### 3. 数据库操作重试
- **场景**: 数据库死锁或连接超时
- **示例**: 事务冲突导致写入失败，需要重试
- **价值**: 保证数据一致性

### 4. 文件操作重试
- **场景**: 文件被占用或临时不可访问
- **示例**: 读写文件时遇到权限错误，需要重试
- **价值**: 提高文件操作的可靠性

### 5. 外部依赖调用
- **场景**: 外部服务暂时不可用
- **示例**: 微服务调用超时，需要重试
- **价值**: 提高分布式系统的容错能力

## 核心优势

### ✅ 优点

1. **上下文累积**: 每次失败记录到 debug.md，下一轮重试加载历史上下文
2. **错误分析**: 自动分析错误类型，生成假设（Hypothesis），帮助 AI Agent 理解失败原因
3. **结构化日志**: 使用固定格式（现象、复现、猜想、验证、修复、进展）记录调试信息
4. **自动终止**: 超过重试限制后自动退出，避免无限循环
5. **渐进式延迟**: 每次重试之间增加延迟时间，避免资源浪费
6. **易于扩展**: 可以自定义重试策略、错误分析逻辑

### ⚠️ 注意事项

1. **非幂等操作**: 如果操作不是幂等的（如银行转账），需要额外的防重复机制
2. **资源消耗**: 多次重试会消耗更多资源（时间、内存、API 配额）
3. **重试限制**: 需要合理设置最大重试次数，避免无限重试
4. **上下文膨胀**: debug.md 可能随着重试次数增长而变得很大，需要限制输出长度
5. **人工干预**: 超过重试限制后需要人工介入修改任务定义或测试方法

## 适用条件

- ✅ 操作可能出现瞬态故障（如网络超时、资源临时不可用）
- ✅ 操作是幂等的，或可以通过状态检查避免重复执行
- ✅ 失败后有改进空间（如 AI Agent 可以根据错误信息调整策略）
- ✅ 可以容忍一定的延迟（重试需要时间）
- ✅ 有明确的失败标准和成功标准

## 不适用场景

- ❌ 非幂等操作且无法通过状态检查避免重复（如支付、发送邮件）
- ❌ 永久性故障（如文件不存在、权限不足、逻辑错误）
- ❌ 实时性要求极高的操作（重试会引入延迟）
- ❌ 错误无法通过重试解决（如语法错误、配置错误）

## 关键概念

### 瞬态故障（Transient Fault）
- **定义**: 临时性的、可能自行恢复的故障
- **示例**: 网络超时、数据库死锁、资源临时不可用
- **特点**: 重试后可能成功

### 永久性故障（Permanent Fault）
- **定义**: 持久性的、无法通过重试解决的故障
- **示例**: 文件不存在、权限不足、逻辑错误
- **特点**: 重试无效，需要修复根本原因

### 幂等性（Idempotency）
- **定义**: 多次执行相同操作的结果与执行一次相同
- **示例**: GET 请求、文件覆盖写入
- **重要性**: 重试模式要求操作具有幂等性

### 指数退避（Exponential Backoff）
- **定义**: 每次重试之间的延迟时间呈指数增长
- **公式**: `delay = baseDelay * (2 ^ attempt)`
- **优点**: 避免频繁重试导致的资源浪费

### 上下文累积（Context Accumulation）
- **定义**: 每次失败记录调试信息，下一轮重试加载历史上下文
- **价值**: 让 AI Agent 从失败中学习，避免重复相同错误
- **实现**: 使用 debug.md 记录失败历史

## Rick CLI 中的重试流程

### 执行循环
1. **生成测试脚本**（一次性，在重试循环外）
2. **重试循环**（最多 5 次）：
   - 加载 debug.md（历史失败上下文）
   - 调用 Claude Code 执行任务
   - 运行测试脚本
   - 如果通过：Git commit + 标记 done
   - 如果失败：记录到 debug.md，继续重试
3. **超过限制**：退出进程，需人工修改 task.md

### debug.md 格式
```markdown
## debug1: Task task1 - Attempt 1/5

**现象 (Phenomenon)**:
- 测试未通过：期望文件 output.txt 存在，但未找到

**复现 (Reproduction)**:
- Task: 创建输出文件
- Goal: 生成 output.txt 文件
- Attempt: 1 of 5

**猜想 (Hypothesis)**:
- 文件或资源不存在 - 可能是路径错误或文件未创建

**验证 (Verification)**:
- Review the output below
- Check if files were created/modified as expected
- Verify test script logic is correct

**修复 (Fix)**:
- Will retry with updated context
- Agent should learn from this failure

**进展 (Progress)**:
- Status: 🔄 重试中 - Attempt 1/5

**输出 (Output)**:
```
File not found: output.txt
```
```

## 实际效果

在 Rick CLI 项目中：
- **重试次数**: 默认最多 5 次
- **成功率**: 约 80% 的失败任务在 3 次重试内成功
- **调试信息**: 每次失败自动记录到 debug.md，便于人工分析
- **自动终止**: 超过 5 次后自动退出，避免无限循环
