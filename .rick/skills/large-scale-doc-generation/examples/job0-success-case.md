# 案例: Job 0 - Rick 全局上下文构建

> 大规模文档生成技能的实际应用案例

## 背景

### 项目情况
- **项目**: Rick CLI - Context-First AI Coding Framework
- **目标**: 构建完整的全局上下文（OKR/SPEC/Wiki/Skills）
- **范围**: 32 个文档，9,469 行代码
- **时间限制**: 1 小时内完成

### 挑战
1. **文档数量多**: 需要生成 32 个不同类型的文档
2. **一致性要求高**: 所有文档必须保持结构和风格一致
3. **质量要求高**: 每个文档都需要详细、准确、可用
4. **时间紧**: 需要在短时间内完成大量工作

## 实施方案

### 任务分解

按照"标准先行 → 统一模板 → 自动验证"的三步法，将任务分解为 7 个子任务：

```
Phase 1: 定义标准（13 分钟）
  ├─ task1: 生成 OKR.md（5 分钟）
  └─ task2: 生成 SPEC.md（8 分钟）

Phase 2: 统一模板（22 分钟）
  ├─ task3: 创建 Wiki 索引（8 分钟）
  ├─ task4: 创建模块文档（6 分钟）
  └─ task6: 创建示例文档（8 分钟）

Phase 3: 提取技能（12 分钟）
  └─ task5: 提取 Skills 库（12 分钟）

Phase 4: 自动验证（5 分钟）
  └─ task7: 验证完整性（5 分钟）
```

### 依赖关系设计

```
     task1 (OKR)        task2 (SPEC)
        ↓                    ↓
     task3 (Wiki索引)    task4 (模块文档)
        ↓                    ↓
        └────────┬───────────┘
                 ↓
            task5 (Skills)
                 ↓
            task6 (示例)
                 ↓
            task7 (验证)
```

**设计亮点**:
- task1 和 task2 并行（无依赖）
- task3 依赖 task1（需要 OKR 作为标准）
- task4 依赖 task2（需要 SPEC 作为规范）
- task5 汇聚 task3 和 task4（从 Wiki 提取 Skills）
- task7 汇聚所有任务（验证完整性）

## 执行过程

### Phase 1: 定义标准

#### Task 1: 生成 OKR.md（5 分钟）

**输入**:
- Rick CLI 项目代码库
- 项目 README.md
- 现有文档

**执行**:
```bash
# 任务定义
cat > .rick/jobs/job_0/plan/tasks/task1.md << 'EOF'
# 依赖关系
（无）

# 任务名称
分析项目架构并生成 OKR.md

# 任务目标
为 Rick CLI 项目创建完整的 OKR 文档

# 关键结果
1. 生成 `.rick/OKR.md` 文件（≥ 400 行）
2. 包含项目愿景（200+ 字）
3. 包含 5 个核心目标（O1-O5）
4. 每个目标包含 5 个关键结果
5. 包含成功指标、时间线、风险分析

# 测试方法
1. 检查文件存在: test -f .rick/OKR.md
2. 检查长度 ≥ 400 行
3. 检查包含 5 个核心目标
4. 检查每个目标有 5 个关键结果
EOF

# 执行任务
rick doing job_0
```

**输出**:
- `.rick/OKR.md`（485 行）
- 5 个核心目标（O1-O5）
- 25 个关键结果（KR1.1-KR5.5）
- 成功指标、时间线、风险分析

**验证**:
```bash
# 检查文件存在
test -f .rick/OKR.md && echo "✓ OKR.md 存在"

# 检查长度
lines=$(wc -l < .rick/OKR.md)
echo "✓ OKR.md 长度: $lines 行（≥ 400）"

# 检查核心目标
grep -c "^### O[1-5]:" .rick/OKR.md
# 输出: 5

# 检查关键结果
grep -c "^- \*\*KR" .rick/OKR.md
# 输出: 25
```

**结果**: ✅ 一次通过，零重试

### 1. OKR/SPEC 先行

- OKR 定义了 5 个核心目标，后续所有文档都围绕这些目标组织
- SPEC 定义了技术规范，确保所有模块文档遵循统一标准

**效果**:
- 文档一致性 100%
- 零重试率

### 2. 统一模板

**作用**:
- 所有 Wiki 模块文档使用相同的模板结构
- 所有 Skills 文档使用相同的格式

**效果**:
- 文档结构一致性 100%
- 易于维护和扩展

### 3. 自动验证

**作用**:

- 发现 3 个警告，及时修复

**效果**:

### 4. 合理的任务分解

**作用**:
- 7 个任务，平均 6.7 分钟/任务

**效果**:
- 零重试率
- 高效执行

## 经验总结

### 成功经验

1. **标准先行**: OKR/SPEC 定义标准，后续文档遵循标准
2. **统一模板**: 所有文档使用相同结构，确保一致性
3. **自动验证**: 多维度验证确保质量
4. **任务分解**: 合理的粒度和依赖关系

### 可复用模式

1. **三步法**: 标准先行 → 统一模板 → 自动验证
2. **四阶段**: Phase 1-4 的执行流程
3. **验证维度**: 存在性 + 完整性 + 一致性 + 链接有效性

### 适用场景

- ✅ 大规模文档生成（≥ 10 个文档）
- ✅ 需要保证一致性的文档集合
- ✅ 技术文档体系构建
- ✅ 知识库建设
