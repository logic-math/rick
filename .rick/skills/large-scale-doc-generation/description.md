# 大规模文档生成技能

> 基于 Context-First 理念的大规模文档生成方法论

## 概述

大规模文档生成技能是一种系统化的文档工程方法，通过"标准先行 → 统一模板 → 自动验证"的三步法，实现高质量、高一致性的大规模文档生成。

**核心理念**: Context-First，即通过建立完整的上下文（OKR/SPEC），确保后续文档生成的一致性和质量。

## 使用场景

### 适用场景 ✅

1. **技术文档体系构建**
   - 项目 OKR、技术规范、架构文档
   - API 文档、开发指南、用户手册
   - Wiki 知识库、Skills 技能库

2. **知识库建设**
   - 企业知识库
   - 技术博客系列
   - 教程和课程体系

3. **标准化文档生成**
   - 需要统一格式和结构的文档
   - 需要保证一致性的文档集合
   - 需要自动化验证的文档

### 不适用场景 ❌

1. **单一文档生成**: 如果只需要生成 1-2 个文档，使用本技能过于复杂
2. **探索性写作**: 如果文档内容和结构尚未确定，不适合使用本技能
3. **创意性内容**: 如果需要高度创意和个性化的内容，本技能可能限制创造力

## 技能优势

### 1. 高一致性

通过 OKR/SPEC 先行，确保所有文档遵循统一标准：

- **结构一致**: 所有文档使用相同的章节结构
- **术语一致**: 关键术语在所有文档中保持一致
- **风格一致**: 写作风格和格式保持一致

**实践数据**（Job 0）:
- 生成 24 个 Wiki 文档，结构完全一致
- 8 个 Skills 文档，格式完全统一
- 验证通过率 103%（56/54）

### 2. 高质量

通过自动化验证，确保文档质量：

- **完整性验证**: 所有必需章节都存在
- **内容验证**: 关键词覆盖、代码示例数量
- **链接验证**: 所有内部链接有效

**实践数据**（Job 0）:
- 54 项自动化检查
- 0 个关键问题
- 3 个警告（不影响使用）

### 3. 高效率

通过任务分解和并行执行，提高生成效率：

- **并行生成**: 无依赖的文档可以并行生成
- **增量生成**: 只生成变化的部分
- **自动化**: 减少人工干预

**实践数据**（Job 0）:
- 7 个任务，47 分钟完成
- 生成 9,469 行文档
- 零重试率

### 4. 可维护性

通过统一模板和自动验证，降低维护成本：

- **易于更新**: 修改模板即可批量更新
- **易于扩展**: 添加新文档只需遵循模板
- **易于验证**: 自动化验证脚本

## 技能局限

### 1. 前期投入

需要投入时间设计 OKR/SPEC 和模板：

- **时间成本**: 需要 1-2 小时设计标准和模板
- **学习成本**: 需要理解 Context-First 理念
- **适用性**: 只有在文档数量 ≥ 10 时才值得投入

### 2. 灵活性限制

统一模板可能限制个性化：

- **结构固定**: 所有文档必须遵循相同结构
- **风格统一**: 难以实现个性化风格
- **创意受限**: 不适合需要高度创意的内容

### 3. 依赖上下文

需要完整的上下文才能发挥效果：

- **OKR 依赖**: 需要明确的项目目标
- **SPEC 依赖**: 需要详细的技术规范
- **代码依赖**: 需要实际代码作为参考

## 实现步骤

### Phase 1: 定义标准（20-30 分钟）

生成标准文档，定义项目目标和技术规范。

**输入**:
- 项目代码库
- 项目需求
- 现有文档（如有）

**输出**:
- `OKR.md`: 项目目标和关键结果（400+ 行）
- `SPEC.md`: 技术规范和架构设计（800+ 行）

**关键要点**:
- OKR 应包含 3-5 个核心目标
- 每个目标应有 3-5 个可衡量的关键结果
- SPEC 应包含架构设计、技术选型、开发规范

**示例**（Job 0）:
```markdown
# OKR.md 结构
- 项目愿景（200+ 字）
- 5 个核心目标（O1-O5）
- 25 个关键结果（5 KRs per objective）
- 成功指标、时间线、风险分析
```

### Phase 2: 统一模板（30-40 分钟）

基于 OKR/SPEC，生成结构化文档。

**输入**:
- `OKR.md`（Phase 1 输出）
- `SPEC.md`（Phase 1 输出）
- 文档列表（需要生成的文档）

**输出**:
- Wiki 索引（`wiki/index.md`）
- 模块文档（`wiki/modules/*.md`）
- 教程文档（`wiki/tutorials/*.md`）

**关键要点**:
- 所有文档使用统一模板
- 模板应包含：概述、核心类型、主要函数、使用示例、常见问题
- 文档间应有交叉引用

**示例**（Job 0）:
```markdown
# 模块文档模板
## 概述
## 核心类型
## 主要函数
## 使用示例
## 常见问题
## 相关模块
```

### Phase 3: 提取技能（20-30 分钟）

从文档和代码中提取可复用技能。

**输入**:
- Wiki 文档（Phase 2 输出）
- 项目代码库
- SPEC 规范

**输出**:
- Skills 索引（`skills/index.md`）
- 技能文档（`skills/*/description.md`）
- 实现文档（`skills/*/implementation.md`）
- 示例文档（`skills/*/examples/*.md`）

**关键要点**:
- 每个技能应包含：描述、实现、示例
- 技能应从实际代码中提取
- 技能应具有可复用性

**示例**（Job 0）:
```markdown
# 技能结构
skills/
├── dag-topological-sort/
│   ├── description.md       # 技能描述（200+ 字）
│   ├── implementation.md    # 实现细节（代码示例）
│   └── examples/            # 实际应用案例
│       └── rick-task-scheduling.md
```

### Phase 4: 自动验证（10-15 分钟）

编写验证脚本，确保文档完整性和一致性。

**输入**:
- 所有生成的文档（Phase 1-3 输出）
- 验证标准（从 OKR/SPEC 提取）

**输出**:
- 验证脚本（`validate_context.sh`）
- 验证报告（`validation_report.md`）
- 总结报告（`summary.md`）

**关键要点**:
- 验证维度：存在性、完整性、一致性、链接有效性
- 自动化执行，输出详细报告
- 发现问题后自动修复或提示

**示例**（Job 0）:
```bash
# 验证脚本结构
1. 核心文档验证（5 项）
2. Skills 库验证（24 项）
3. Wiki 知识库验证（11 项）
4. 文档链接验证（3 项）
5. 代码示例验证（1 项）
6. 关键词覆盖验证（8 项）
7. 一致性验证（4 项）

总计：54 项检查
```

## 最佳实践

### 1. OKR/SPEC 设计

**DO ✅**:
- OKR 应包含 3-5 个核心目标
- 每个目标应有 3-5 个可衡量的关键结果
- SPEC 应包含架构设计、技术选型、开发规范
- 使用 SMART 原则（Specific, Measurable, Achievable, Relevant, Time-bound）

**DON'T ❌**:
- 不要过于简单（< 200 行）
- 不要过于详细（> 2000 行）
- 不要缺少关键章节（目标、架构、规范）
- 不要忽略时间线和成功指标

### 2. 模板设计

**DO ✅**:
- 模板应简单明了，易于遵循
- 模板应包含必需章节和可选章节
- 模板应支持交叉引用
- 模板应考虑可扩展性

**DON'T ❌**:
- 不要过于复杂（> 10 个章节）
- 不要过于简单（< 3 个章节）
- 不要缺少示例
- 不要忽略常见问题（FAQ）

### 3. 验证设计

**DO ✅**:
- 验证应自动化执行
- 验证应覆盖多个维度（存在性、完整性、一致性）
- 验证应输出详细报告
- 验证应支持自动修复

**DON'T ❌**:
- 不要依赖人工验证
- 不要只验证存在性
- 不要忽略链接有效性
- 不要忽略一致性验证
