# DAG 任务分解方法

## 技能描述

DAG（有向无环图）任务分解方法是一种将复杂任务分解为多个小任务，并通过依赖关系组织这些任务的方法。通过合理的 DAG 设计，可以实现任务的并行执行、依赖管理和错误隔离。在 Job 1 中，我们设计了一个教科书级别的 DAG 任务图，实现了零重试率的高效执行。

**核心理念**: 将复杂任务分解为多个小任务，通过依赖关系确保任务的执行顺序和上下文完整性。

## 适用场景

- **复杂项目分解**: 将大型项目分解为多个可管理的小任务
- **文档生成**: 需要生成大量相互关联的文档
- **代码重构**: 需要分步骤进行的代码重构
- **系统迁移**: 需要分阶段进行的系统迁移
- **数据处理流水线**: 需要多步骤处理的数据流水线

## 实现模式

### 方案1: 三层 DAG 结构（基础 → 生产 → 验证）

**核心思想**: 将任务分为三层：基础层、生产层、验证层

**结构**:
```
         基础层
         /    \
    生产层1  生产层2 (并行)
        \    /
       生产层3 (汇聚)
          |
       验证层
```

**优点**:
- 结构清晰，易于理解
- 便于并行化
- 易于错误隔离

**缺点**:
- 可能不适用于所有场景
- 需要仔细设计层次划分

**Job 1 示例**:

```
         task1 (基础层：目录结构)
         /    \
     task2    task3 (基础层：架构和流程文档，并行)
        \    /
        task4 (生产层：模块文档，汇聚)
          |
     task5, task6 (生产层：专题文档，串行)
          |
     task7, task8 (生产层：专题文档，串行)
          |
        task9 (验证层：验证和完善)
```

**关键设计点**:
1. **task1 作为基础**: 创建目录结构，为后续任务提供基础
2. **task2 和 task3 并行**: 架构和流程文档可以并行生成
3. **task4 作为汇聚点**: 依赖 task2 和 task3，确保基础完成后再批量生产
4. **task5-8 串行**: 专题文档依次生成，每个任务都能利用前序任务的输出
5. **task9 作为验证**: 依赖所有前序任务，进行全面验证

### 方案2: 扇出-扇入结构（Fan-out Fan-in）

**核心思想**: 一个基础任务扇出到多个并行任务，然后扇入到一个汇聚任务

**结构**:
```
      task1 (基础)
     /  |  |  \
task2 task3 task4 task5 (并行)
     \  |  |  /
      task6 (汇聚)
```

**优点**:
- 最大化并行度
- 适合大量独立任务
- 执行效率高

**缺点**:
- 需要确保并行任务真正独立
- 汇聚任务可能复杂

**示例场景**: 模块文档生成

```markdown
# task1: 创建目录结构
依赖: 无

# task2-8: 生成各个模块文档（并行）
依赖: task1

# task9: 验证所有模块文档
依赖: task2, task3, task4, task5, task6, task7, task8
```

### 方案3: 流水线结构（Pipeline）

**核心思想**: 任务按照流水线顺序串行执行，每个任务的输出是下一个任务的输入

**结构**:
```
task1 → task2 → task3 → task4 → task5
```

**优点**:
- 结构简单，易于理解
- 上下文传递清晰
- 适合有明确顺序的任务

**缺点**:
- 无法并行化
- 执行效率较低

**示例场景**: 渐进式文档生成

```markdown
# task1: 创建目录结构
输出: wiki/ 目录

# task2: 编写架构文档
输入: wiki/ 目录
输出: architecture.md

# task3: 编写模块文档
输入: architecture.md
输出: modules/*.md

# task4: 编写专题文档
输入: architecture.md, modules/*.md
输出: dag-engine.md, prompt-system.md

# task5: 验证文档
输入: 所有文档
输出: 验证报告
```

## 最佳实践

### 1. 合理划分任务粒度

**原则**: 每个任务的预计执行时间为 5-10 分钟

**理由**:
- 太小（< 5 分钟）: 任务过于琐碎，管理开销大
- 太大（> 10 分钟）: 任务复杂度高，失败风险大

**实践**:
1. 评估任务的工作量
2. 将大任务分解为多个小任务
3. 确保每个任务都有明确的输入和输出

**示例**:
- ✅ **合理粒度**: "创建 7 个模块文档"（预计 10 分钟）
- ❌ **粒度过小**: "创建 cmd.md 的标题"（预计 30 秒）
- ❌ **粒度过大**: "创建所有 Wiki 文档"（预计 2 小时）

### 2. 识别并行化机会

**原则**: 无依赖关系的任务应该并行执行

**理由**:
- 提高执行效率
- 缩短总执行时间
- 充分利用资源

**实践**:
1. 分析任务之间的依赖关系
2. 识别可以并行执行的任务
3. 在 DAG 设计中体现并行关系

**示例**:
```
# 可以并行
task2: 编写架构文档
task3: 编写流程文档
（这两个任务没有依赖关系）

# 不能并行
task4: 编写模块文档（依赖 task2 和 task3）
task5: 编写专题文档（依赖 task4）
```

### 3. 使用汇聚点设计

**原则**: 在阶段性成果完成后设置汇聚点

**理由**:
- 确保阶段性成果的完整性
- 便于错误检测和修复
- 提供清晰的里程碑

**实践**:
1. 识别项目的关键阶段
2. 在每个阶段结束时设置汇聚任务
3. 汇聚任务负责验证阶段性成果

**示例**:
```
         task1 (基础)
         /    \
     task2    task3 (并行基础)
        \    /
        task4 (汇聚点：验证基础完成)
          |
     task5-8 (批量生产)
          |
        task9 (汇聚点：验证全部完成)
```

### 4. 明确任务依赖关系

**原则**: 每个任务的依赖关系应该清晰、明确

**理由**:
- 确保任务的执行顺序正确
- 避免循环依赖
- 便于 DAG 拓扑排序

**实践**:
1. 在 task.md 中明确列出依赖的任务
2. 使用任务 ID 而非任务名称
3. 避免隐式依赖

**示例** (task.md):
```markdown
# 依赖关系
task2, task3

# 任务名称
编写模块文档

# 任务目标
基于 task2 的架构文档和 task3 的流程文档，编写 7 个核心模块的详细文档。
```

### 5. 避免循环依赖

**原则**: DAG 中不应该存在循环依赖

**理由**:
- 循环依赖导致任务无法执行
- 违反 DAG 的定义
- 难以调试和修复

**实践**:
1. 在设计 DAG 时检查循环依赖
2. 使用拓扑排序验证 DAG 的有效性
3. 如果存在循环依赖，重新设计任务分解

**示例**:
```
# ❌ 循环依赖
task1 → task2 → task3 → task1

# ✅ 正确的 DAG
task1 → task2 → task3 → task4
```

### 6. 设计清晰的输入输出

**原则**: 每个任务应该有明确的输入和输出

**理由**:
- 便于理解任务的作用
- 便于验证任务的完成
- 便于设计依赖关系

**实践**:
1. 在任务定义中明确输入和输出
2. 输入应该来自依赖任务的输出
3. 输出应该可以被后续任务使用

**示例**:
```markdown
# task2: 编写架构文档

**输入**:
- task1 的输出: wiki/ 目录结构
- 项目源码: internal/ 目录

**输出**:
- wiki/architecture.md（架构概览文档）
```

## 常见陷阱

### ❌ 陷阱1: 过度串行化

**问题**: 将可以并行的任务设计为串行执行

**后果**: 执行效率低，总执行时间长

**避免方法**:
- 仔细分析任务之间的依赖关系
- 识别可以并行执行的任务
- 在 DAG 设计中体现并行关系

**示例**:
```
# ❌ 过度串行化
task1 → task2 → task3 → task4 → task5

# ✅ 合理并行化
         task1
         /    \
     task2    task3
        \    /
        task4
          |
        task5
```

### ❌ 陷阱2: 过度并行化

**问题**: 将有隐式依赖的任务设计为并行执行

**后果**: 任务之间可能产生冲突或不一致

**避免方法**:
- 仔细分析任务之间的隐式依赖
- 只有在确实无依赖时才并行化
- 使用汇聚点设计确保阶段性成果的完整性

**示例**:
```
# ❌ 过度并行化（task3 和 task4 有隐式依赖）
         task1
       /  |  \
   task2 task3 task4

# ✅ 正确设计
         task1
         /    \
     task2    task3
        \    /
        task4
```

### ❌ 陷阱3: 任务粒度不当

**问题**: 任务粒度过大或过小

**后果**:
- 过大: 任务复杂度高，失败风险大
- 过小: 任务过于琐碎，管理开销大

**避免方法**:
- 每个任务的预计执行时间控制在 5-10 分钟
- 将大任务分解为多个小任务
- 合并过小的任务

### ❌ 陷阱4: 缺少汇聚点

**问题**: 在阶段性成果完成后没有设置汇聚点

**后果**: 无法验证阶段性成果的完整性，问题可能累积

**避免方法**:
- 在每个阶段结束时设置汇聚任务
- 汇聚任务负责验证阶段性成果
- 提供清晰的里程碑

### ❌ 陷阱5: 循环依赖

**问题**: DAG 中存在循环依赖

**后果**: 任务无法执行，系统报错

**避免方法**:
- 在设计 DAG 时检查循环依赖
- 使用拓扑排序验证 DAG 的有效性
- 重新设计任务分解以消除循环依赖

## 测试建议

### 1. DAG 有效性验证

验证 DAG 是否有效（无循环依赖）。

**方法**: 使用拓扑排序算法（Kahn 算法）

**示例代码**:
```go
func TopologicalSort(tasks []Task) ([]string, error) {
    // 计算入度
    inDegree := make(map[string]int)
    for _, task := range tasks {
        if _, exists := inDegree[task.ID]; !exists {
            inDegree[task.ID] = 0
        }
        for _, dep := range task.Dependencies {
            inDegree[dep]++
        }
    }

    // Kahn 算法
    queue := []string{}
    for id, degree := range inDegree {
        if degree == 0 {
            queue = append(queue, id)
        }
    }

    result := []string{}
    for len(queue) > 0 {
        current := queue[0]
        queue = queue[1:]
        result = append(result, current)

        // 移除边
        for _, task := range tasks {
            if task.ID == current {
                for _, dep := range task.Dependencies {
                    inDegree[dep]--
                    if inDegree[dep] == 0 {
                        queue = append(queue, dep)
                    }
                }
            }
        }
    }

    // 检查是否所有任务都被访问
    if len(result) != len(tasks) {
        return nil, fmt.Errorf("cycle detected in DAG")
    }

    return result, nil
}
```

### 2. 依赖关系验证

验证每个任务的依赖关系是否正确。

**检查项**:
- 依赖的任务是否存在
- 是否存在循环依赖
- 是否存在孤立任务（没有前驱也没有后继）

### 3. 并行化验证

验证并行任务是否真正独立。

**检查项**:
- 并行任务是否有隐式依赖
- 并行任务是否会产生冲突
- 并行任务的输出是否会被后续任务正确使用

## 成功案例

**Job 1: Wiki 文档创建**

**DAG 结构**:
```
         task1 (基础)
         /    \
     task2    task3 (并行基础)
        \    /
        task4 (汇聚)
          |
     task5, task6 (串行专题)
          |
     task7, task8 (串行专题)
          |
        task9 (验证)
```

**执行结果**:
- 任务数量: 9 个
- 零重试率: 100% (9/9)
- 执行时长: ~2 小时
- 并行化收益: task2 和 task3 并行执行，节省约 10 分钟

**设计亮点**:
1. **合理的并行化**: task2 和 task3 可并行执行
2. **汇聚点设计**: task4 依赖 task2 和 task3，确保基础完成后再批量生产
3. **串行专题任务**: task5-8 串行执行，每个任务都能利用前序任务的输出
4. **最终验证**: task9 依赖所有前序任务，进行全面验证

**经验总结**:
- ✅ 合理的并行化可以提高效率
- ✅ 汇聚点设计确保阶段性成果的完整性
- ✅ 串行任务可以利用前序任务的输出作为上下文
- ⚠️ task5-8 可以进一步并行化（它们之间没有依赖关系）

这次成功充分验证了 DAG 任务分解方法的有效性。
